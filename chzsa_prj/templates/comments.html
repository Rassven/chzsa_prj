Общее:

Надежность в технике. Термины и определения.

- Отслеживание состояния подразумевает логирование каждой операции изменения модели (а не текущее состояние модели). Идеальной будет структура: деталь.параметры (неизменны) + деталь.история (все этапы жизненного цикла для этого к моделям узлов добавлено JSON-поле "work_history").

- Данные по новым компонентам/деталям не вводятся вручную, а берутся (должны браться) из БД (на худой конец некоего файла с описанием поставляемой единицы (спецификация)) предоставленного производителем/пользователем/сервисной организацией (распределены по времени).

- "Образ" машины складывается потом из отдельных её деталей (а не наоборот), тогда уже можно что-то прогнозировать.

- Проблема с одновременной заменой двух и более деталей (а по задумке это и следует планировать для ТО), предложенная структура подразумевает замену по одной детали за раз (и подразумевается ручной ввод вместо работы со списками).

- Обновление данных от организаций делаются только в полях, где конкретная организация прописана Владельцем поля.

Нет разграничения "владельца поля" (право изменять). Еще модели содержат поля от разных "правообладателей". Нужны таблицы от каждого из участников (пооперационно, с привязкой ко времени и детальным описанием действий), которые потом можно свести в общую таблицу.

Для неавторизованного пользователя поиск по номеру машины следует заменить на поиск по модели. Просмотр комплектаций без вывода серийных номеров.

Команда на формирование сводной Excel-таблицы доступна только менеджеру производителя. Для сторонних организаций следует формировать "профильные" таблицы.

Сервис не должен формировать модели "вручную", информация формируется конкретной организацией. Иначе это приведет к рассогласованию с фактическим состоянием. Он должен лишь аккумулировать информацию, поступающую из разных источников.

Предложенная "машино-центричная" () модель не подходит для решаемой задачи, нужна "организационно-центричная" модель (). Где пользователи, узлы и сервисы - производные от конкретных фирм и привязаны к ним.


По модели Машина (список узлов предоставляется производителем, отдельное описание ключевых не целесообразно):

Узел должен включать и модель и свой серийный номер, а так же характеристики и историю.

Торговые отношения и их подробности целесообразно вынести в отдельную модель.

Разделение на "Клиент" и "Конечный пользователь" должно строиться вне модели Машины. Если вообще необходимо их упоминать.

По предложенной схеме в сервисе будут храниться только данные по проданным единицам.
